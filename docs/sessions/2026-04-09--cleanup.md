traininbg is quite slow. Please list some ways to speed it up/ Are there any ways to paralelize it?

```
• Yes. There are several ways to speed it up, and some parts can be parallelized, but not all of them equally well.

  The current training cost is dominated by two things:

  - generating Julia feature vectors for all samples in code/v001/packages/shared/src/julia/training-samples.ts
  - the SOM training loop in code/v001/packages/shared/src/som/train.ts, which for every sample scans all cells for the BMU and then updates all cells again

  So rough cost is close to:

  - sample generation: sampleCount * featureWidth * featureHeight * juliaIterations
  - training: rounds * sampleCount * cellCount * featureLength

  For large maps and large feature vectors, that gets expensive fast.

  Best speedups with low conceptual risk

  - Reduce work per sample.
    Lower featureWidth, featureHeight, trainingJuliaIterations, trainingRounds, or map size. This is the simplest lever and gives immediate linear or quadratic wins depending on what you reduce.
  - Cache generated training samples.
    Right now the worker calls generateTrainingSamples(settings) every training run in code/v001/packages/web/src/workers/training-worker.ts:49. If you retrain with the same sample-generation settings but only change SOM parameters, you could
    reuse the same sample matrix instead of recomputing all Julia images.
  - Avoid object-heavy data structures.
    The current code uses arrays of sample objects and cell objects. A flatter typed-array layout for prototype vectors and sample vectors would reduce GC pressure and improve cache locality.
  - Precompute shuffled orders once if settings/seed are unchanged.
    createDeterministicShuffledIndices(...) is called each round in code/v001/packages/shared/src/som/train.ts:59. That is not the biggest cost, but it is avoidable.

  Biggest algorithmic speedups

  - Stop updating every cell for every sample.
    In code/v001/packages/shared/src/som/train.ts:87-104, once the BMU is found, every cell is visited and updated. With a Gaussian neighborhood, many far-away cells contribute almost nothing. You can use a radius cutoff and update only cells
    within a bounding neighborhood. That is one of the best practical speedups.
  - Faster BMU search.
    BMU search is a full scan across all cells for every sample. That is exact but expensive. Approximate nearest-neighbor methods or coarse-to-fine search could help, though they change training behavior somewhat.
  - Batch SOM instead of pure online SOM.
    Your current implementation is classic online BMU-driven SOM. Batch variants can sometimes be implemented more efficiently and can parallelize better, but that is a model change.

  Parallelization

  Yes, some parts parallelize well.

  - Training sample generation parallelizes very well.
    Each Julia sample is independent. This is the cleanest place to parallelize. You could split sample indices across multiple workers and generate feature vectors in parallel, then merge them. This is probably the safest and highest-value
    parallelization target.
  - Mandelbrot/Julia rendering parallelizes well too, if needed.
    Those are embarrassingly parallel per pixel.
  - BMU distance computation can be parallelized.
    For one sample, you can split the cells across workers, let each worker find a local best BMU, then reduce to the global best. This is feasible, but the overhead may be high unless maps are large.
  - Cell updates can be parallelized only in a controlled way.
    After the BMU is known, each cell update is independent for that one sample, so in theory the update pass can be split across workers. But the synchronization cost is much higher than for sample generation, and you need shared memory or
    careful copying to make it worthwhile.

  What does not parallelize cleanly

  - The online training sequence across samples is inherently ordered.
    Sample n+1 depends on the weights after sample n. So you cannot simply train different samples in parallel and combine them without changing the algorithm.

  That means:

  - parallelize across pixels or samples during sample generation: good
  - parallelize inside one BMU/update step: possible but more complex
  - parallelize the whole online training loop across samples: not clean unless you change algorithms

  Best practical order for optimization

  1. Add caching for generated training samples.
  2. Add neighborhood cutoff so far cells are skipped during updates.
  3. Flatten data into typed arrays for prototypes/samples.
  4. Parallelize Julia sample generation across workers.
  5. Only then consider parallel BMU/update computation.

  If you want, I can turn this into a concrete optimization plan for this codebase with expected impact and implementation difficulty.
```
