• Yes. I see the likely root cause.

  The panning logic is implemented inside an effect that depends on viewport in both canvases:

  - code/v001/packages/web/src/canvas/mandelbrot-overview-canvas.tsx
  - code/v001/packages/web/src/canvas/julia-viewer-canvas.tsx

  That matters because during a drag, mousemove calls setViewport(...). As soon as viewport changes, the effect re-runs, and its cleanup executes first. In that cleanup, the code does this:

  - clears dragStateRef.current
  - removes the event listeners
  - resets the cursor

  So the drag gesture gets torn down immediately after the first viewport update. In practice that makes panning feel broken or nonfunctional, even though the first movement may technically apply a tiny update.

  So the bug is not the viewport math itself. The bug is the event lifecycle:

  1. mousedown starts drag
  2. mousemove computes a pan and calls setViewport(...)
  3. component re-renders because viewport changed
  4. effect cleanup runs
  5. drag state is wiped out
  6. the ongoing drag is effectively cancelled

  That is why zoom works fine: wheel interactions are discrete and don’t depend on sustained drag state. Panning needs stable listeners and stable drag state across many mousemove events, but the current effect structure destroys them mid-
  gesture.

  The clean fix will be to keep the pointer event listeners stable and avoid rebuilding them on every viewport change. Typically that means:

  - removing viewport from the drag-listener effect dependencies
  - reading the current viewport through refs or functional state updates instead

  If you want, I can patch it that way next.
