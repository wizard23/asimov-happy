• Yes. If the goal is the actual attracting-cycle period, distance autocorrelation is the wrong primary method.

  A better approach is to detect periodicity in the orbit itself.

  ## What “actual period” means

  For the orbit

  - z_(n+1) = z_n^2 + c

  a point is attracted to a cycle of period p if eventually

  - z_(n+p) ≈ z_n

  for the smallest positive p.

  So the task is:

  - skip the transient
  - inspect the tail of the orbit
  - find the smallest p for which the orbit repeats

  ## Correct practical method

  Use direct orbit repetition detection.

  ### Basic algorithm

  1. Generate orbit points z_0, z_1, ..., z_N
  2. Discard an initial transient prefix
  3. For candidate periods p = 1..Pmax
  4. Check whether the tail satisfies:
      - |z_n - z_(n-p)| < epsilon
        for many consecutive n
  5. Choose the smallest p that passes robustly

  This is much closer to the actual dynamical period.

  ## Why this is better than autocorrelation

  Autocorrelation of distances:

  - loses phase information
  - can confuse harmonics
  - can report repeated radius patterns instead of true cycle repetition

  Direct complex-orbit comparison:

  - uses full complex state
  - matches the mathematical definition of periodicity
  - is much better at identifying the minimal period

  ## What still makes it tricky

  There are real edge cases:

  1. Transients

  - the orbit may wander for many steps before settling

  2. Neutral / boundary regions

  - convergence may be very slow
  - near the Mandelbrot boundary the notion of a clean attracting period may be numerically unstable

  3. Escaping points

  - outside the Mandelbrot set there is no attracting bounded cycle of the kind you want

  So the UI should only claim a period when detection is robust.

  ## Stronger method: periodicity from the tail

  Recommended v1 detector:

  1. Compute orbit up to N steps, e.g. N = 512 or 1024
  2. If orbit escapes, report:
      - no attracting cycle
  3. Otherwise inspect only the last W points, e.g. W = 128
  4. For each candidate period p = 1..Pmax
      - compare z_n with z_(n-p) over the tail window
      - use a tolerance epsilon
      - require that all or almost all comparisons pass
  5. Return the smallest passing p

  That gives you the minimal detected period of the attracting cycle.

  ## Even better method: cluster the tail into cycle points

  Another robust approach:

  1. Compute a long bounded orbit
  2. Take the tail points
  3. Group points that are within tolerance of each other
  4. Count the number of distinct repeating positions in the cycle

  This can work well, but it is a little more implementation-heavy than direct z_n vs z_(n-p) testing.

  ## Best practical spec

  If you want correctness and simplicity, I recommend:

  - only detect period for non-escaping orbits
  - use direct complex-orbit tail comparison
  - report the smallest stable period found
  - if no stable repeat is found, show:
      - period: undetermined

  ## Suggested UI spec

  - Checkbox: Show Attracting Period
  - Optional numeric field: Period Detection Steps
  - Optional numeric field: Max Period
  - Default:
      - off
      - steps 512
      - max period 64

  Displayed result:

  - period = 1
  - period = 3
  - period = 5
  - period: none
  - period: undetermined

  ## Important honesty constraint

  Even with the better method, “actual period” should still be interpreted numerically:

  - exact mathematical certification is harder
  - but for interactive exploration this direct-orbit method is the right way to do it

  - not by autocorrelating scalar distances

• Those numbers are heuristics, not deep constants.

  Why they are there:

  - 32 is just a floor so small N values do not start period testing too early.
  - 128 is just a cap so the detector does not overemphasize a very long tail window and become more expensive than needed for interactive use.

  So:

  - max(32, floor(N * 0.5)) means “discard about half the orbit, but never less than a small warmup.”
  - min(128, floor(N * 0.4)) means “use a decent tail sample, but don’t keep growing it forever.”

  You are right to question whether these should be user-definable.

  My view:

  - Period Detection Steps and Max Detected Period should definitely be user controls.
  - transient discard and tail window usually should not be exposed first, unless we discover the defaults are not robust enough.

  Reason:

  - they are detector-tuning parameters, not user-goal parameters
  - exposing too many controls makes the explorer harder to use
  - most users care about:
      - how long to run
      - what maximum period to search
        not:
      - exact warmup fraction
      - exact tail comparison window

  So I’d treat 32 and 128 as v1 internal defaults, not final truth.

  If we want to make the spec cleaner, I would rewrite that part as:

  - use a transient-discard phase and a tail-comparison window derived from Period Detection Steps
  - initial implementation may use heuristic defaults
  - these are implementation parameters, not user-facing controls in v1

  That would be more honest and less arbitrary-looking.

  So your instinct is right:

  - they are not fundamental
  - they could be exposed later
  - but I would not make them user-facing in v1 unless we already know we need that flexibility.
